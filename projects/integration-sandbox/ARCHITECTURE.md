# 架构决策记录 (ADR)

## ADR-001: 选择方案 B（专业角色 + AI 协调）

**状态:** 已接受
**日期:** 2026-03-16
**决策人:** AI + 用户

---

## 背景

在设计 Integration Sandbox 时，面临两种架构选择：
- 方案 A: 全栈智能角色
- 方案 B: 专业角色 + AI 协调

## 决策

**选择方案 B：专业角色 + AI 协调**

## 理由

### 1. 职责分离原则 (Separation of Concerns)

```
方案 A（全栈角色）          方案 B（专业角色 + AI）
┌─────────────────┐        ┌─────────┐    ┌─────────┐
│ 前端沙箱         │        │前端沙箱  │◄──►│后端沙箱  │
│  (全栈能力)      │        │(专注前端)│ AI │(专注后端)│
│                 │        └────┬────┘ 协调└────┬────┘
│ • 前端开发      │             │              │
│ • 后端开发      │        ┌────┴────┐    ┌────┴────┐
│ • 测试编写      │        │测试沙箱  │    │AI协调器  │
│ • 数据库设计    │        │(专注测试)│    │(专注协调)│
└─────────────────┘        └─────────┘    └─────────┘

问题：                    优势：
• 违反单一职责           • 职责清晰
• 复杂度爆炸              • 易于优化
• 难以专业化              • 可扩展性强
```

### 2. 专业化优势

| 角色 | 专注领域 | 深度优化 |
|------|---------|---------|
| 前端沙箱 | React/Vue/Angular | UI 自动化、组件测试 |
| 后端沙箱 | API/DB/Cache | 性能调优、并发测试 |
| 测试沙箱 | 边界 case/覆盖率 | 智能生成、变异测试 |
| AI 协调器 | 翻译/匹配/协调 | NLP、知识图谱、强化学习 |

### 3. AI 的定位

**AI 不是替代角色，而是增强协作：**

```
传统联调:                    AI 增强联调:
前端 ──► 人工沟通 ◄─── 后端   前端 ──► AI 翻译 ◄─── 后端
  │                      │      │
  ▼                      ▼      ▼
测试（人工编写）           测试（AI 生成 + 验证）

AI 的核心价值：
1. 翻译：前端术语 ↔ 后端术语
2. 匹配：自动识别接口对应
3. 协调：智能调度执行顺序
4. 修复：自动修复简单不匹配
```

### 4. 现实映射

```
现实团队结构:                Sandbox 映射:
┌─────────┐                 ┌─────────┐
│ 前端开发  │  ───────────►  │ 前端沙箱 │
└────┬────┘                 └────┬────┘
     │                           │
┌────┴────┐                 ┌────┴────┐
│ 后端开发  │  ───────────►  │ 后端沙箱 │
└────┬────┘                 └────┬────┘
     │                           │
┌────┴────┐                 ┌────┴────┐
│ 测试工程师│  ───────────►  │ 测试沙箱 │
└─────────┘                 └─────────┘
                                  │
                            ┌────┴────┐
                            │ AI 协调器│  ← 新增
                            └─────────┘

优势：符合现有团队结构，易于理解和采用
```

### 角色的智能增强

**前端沙箱的智能：**
```javascript
class SmartFrontendSandbox {
  // 基础能力：运行前端代码
  
  // 智能增强：
  - 自动识别组件层级
  - 智能生成用户操作路径
  - 预测用户行为模式
  - 自动提取 API 调用模式
}
```

**后端沙箱的智能：**
```python
class SmartBackendSandbox:
    # 基础能力：运行后端服务
    
    # 智能增强：
    - 自动识别 API 契约
    - 智能生成边界测试数据
    - 预测性能瓶颈
    - 自动提取业务规则
```

**测试沙箱的智能：**
```python
class SmartTestSandbox:
    # 基础能力：执行测试用例
    
    # 智能增强：
    - 基于需求生成测试用例
    - 自动识别边界条件
    - 智能变异测试
    - 预测测试覆盖率
```

## 对比总结

| 维度 | 方案 A (全栈角色) | 方案 B (专业 + AI) | 胜出 |
|------|------------------|-------------------|------|
| 职责清晰度 | ❌ 模糊 | ✅ 清晰 | B |
| 专业化程度 | ❌ 浅层 | ✅ 深度 | B |
| 可维护性 | ❌ 差 | ✅ 好 | B |
| 可扩展性 | ❌ 难 | ✅ 易 | B |
| 现实映射 | ❌ 不符合 | ✅ 符合 | B |
| 复杂度 | ❌ 高 | ✅ 可控 | B |
| AI 价值 | ❌ 替代 | ✅ 增强 | B |

## 结论

**方案 B（专业角色 + AI 协调）是更好的选择。**

它：
- 符合软件工程最佳实践
- 映射现实团队结构
- 每个角色可以深度优化
- AI 发挥最大价值（协调而非替代）
- 易于理解和推广
